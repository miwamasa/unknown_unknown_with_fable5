Agent.md フェーズ4の必須ドキュメント。設定・結果・**失敗した試行**をすべて記録する。
実行環境: Linux コンテナ、CPU 4コア（GPUなし）、Python 3.11.15。

## 1. 設計上の簡略化（原典が確認できなかったもの）

- **損失の重み**: MDLM 系は吸収状態拡散の変分下界がマスク位置クロスエントロピーの
  時刻依存重み付きになることを示すが、原典を閲覧できないため**一様重み**で実装
  （`diffsr/model/diffusion.py`）。生成品質には影響しうる。
- **式サンプリング**: Lample & Charton の厳密な一様 unary-binary 木ではなく、
  予算付き再帰構築の簡易版（`tree.py:sample_tree`）。演算子重みは
  `DEFAULT_OP_WEIGHTS`（add/mul=10, sub=5, sin/cos=3, div/pow=2, exp/log/sqrt/neg=1）。
- **Nguyen の式・区間**: 原典（Uy et al. 2011）未確認。サーベイ再掲の慣用値を使用。
  条件付けデータは各問題の慣用区間から `cfg.n_points` 点（慣用の 20 点より多い。
  エンコーダの入力点数を学習時と揃えるため）。
- **gplearn の protected 演算**: div/log/sqrt はゼロ割り等を握り潰す protected 版
  だが、SymPy 変換では通常演算として扱う（記号的一致がやや甘くなる方向）。

## 2. M4: 無条件拡散の検証

**失敗した試行**: 最初のゲートテストは 2000 式 × 6 エポック（96 ステップ）で
パース成功率 **0.37**（ゲート 0.8 不合格）。学習不足が原因と特定。

学習量を増やした結果（tiny1var 語彙、無条件、8000 式）:

| ステップ | パース成功率 |
|---|---|
| 252 | 0.74 |
| 504 | 0.90 |
| 756 | 0.76 |
| 1008 | 0.93 |
| 1260 | 0.90 |

→ 8000 式 × 20 エポックでゲート **0.8 通過**（テストに反映）。
成功率はステップ間で ±0.1 程度揺れる。マスクスケジュールの調整余地あり（未実施）。

## 3. M5: tiny1var 最小 end-to-end（合格ゲート）

設定: `tiny1var`（1変数、演算子 add/sub/mul/sin、L=16、d_model=64、
20,000 式、8 エポック、シード 0）。学習時間 **約3分**（CPU）、最終 loss 0.605。

ゲート結果（held-out 10 式、k=16、記号的一致判定は定数スナップ込み）: **7/10 合格**

| 正解 | 判定 | 予測 |
|---|---|---|
| 2*x0+1 | OK | 2*x0+1 |
| x0²+x0 | OK | x0*(x0+1) |
| 3*sin(x0) | **NG** | sin(sin(x0)) |
| sin(x0)+x0 | OK | x0+sin(x0) |
| x0³ | OK | x0³ |
| x0²−2 | **NG** | x0²+x0−1.98 |
| 2*sin(x0)+1 | **NG** | sin(sin(x0))+1 |
| sin(2*x0) | OK | sin(2*x0) |
| 5*x0−3 | OK | 5*x0−3.0 |
| x0*sin(x0) | OK | x0*sin(x0) |

観察: 失敗3件はいずれも「正しいスケルトンが k=16 のサンプル中に出なかった」
パターン（BFGS や判定の失敗ではない）。k を増やせば解消する（§4）。

## 4. M6: best-of-k 曲線と λ 感度（tiny1var、ゲート10式）

| k | 復元数 | 平均無効生成率 |
|---|---|---|
| 1 | 1/10 | 0.50 |
| 4 | 4/10 | 0.42 |
| 16 | 7/10 | 0.41 |
| 64 | **9/10** | 0.45 |

- **k が実質性能を決める**（フェーズ1で指摘した「未知の未知」#4 を定量確認）。
  拡散モデル単体の 1 サンプル精度は低く、多様な候補の生成＋BFGS＋選択という
  パイプライン全体で機能する。
- 無効生成率は 0.4〜0.5。棄却で処理できる水準だが高い（リスク A4 は「ギリギリ許容」。
  文法制約付きデコードが今後の課題）。
- λ（複雑度正則化）∈ {0, 1e-4, 1e-2} は復元数・平均複雑度とも**無感度**。
  この規模では MSE だけで候補順位が決まる。

## 5. M7: proto3var 学習

### 5.1 失敗した試行: 初期構成が時間予算超過

初期設定（3変数、演算子10種、L=32、d_model=128、PMA、**n_points=128、80,000 式、
6 エポック**）は、データ生成 35 秒に対し 1 エポックが 15〜20 分級と判明
（6 エポックで 90〜120 分 ≫ 60 分予算）。エポック途中で停止し、リスク A2 の
想定対応として **n_points 128→64、n_train 80k→60k、epochs 6→4** に縮小した。

### 5.2 正式構成の結果

コマンド: `python scripts/train.py --config proto3var --out runs/proto3var`（シード 0）

| 項目 | 値 |
|---|---|
| データ生成 | 60,000 問題 / 25.4 秒（事前生成キャッシュ。学習ループ内で式生成はしない） |
| 学習時間 | **753 秒（12.5 分）** ← 60 分予算内 |
| 損失 | 0.873 → 0.659 → 0.629 → 0.616 |

### 5.3 ステップ時間の要因分解（外部レビュー gpt55review_on_M7.md への回答）

学習を5回やり直す代わりに、1 ステップあたりの実測時間（batch=128、
ウォームアップ後20ステップ平均）で要因を直接分解した:

| 構成（proto3var からの変更） | s/step |
|---|---|
| 現行（PMA, n_points=64, d_model=128） | 0.247 |
| encoder=points（PMA なし） | 0.255 |
| n_points=32 | 0.179 |
| **n_points=128** | **0.421** |
| d_model=64 | 0.103 |
| d_model=64 + points + n_points=32 | 0.079 |

結論:
- **支配要因は n_points**（観測点数）。エンコーダの self-attention が O(n²) で効く。
  初期構成が遅かった主因は n_points=128 × n_train=80k × 6 エポックの複合。
- **PMA の有無はほぼ中立**（±3%）。レビューの「PMA を外せ」は本実装では効果なし
  （PMA はデコーダ側 cross-attention を短くする方向に働くため）。
- d_model 128→64 で半減するが、表現力とのトレードオフのため現行は 128 を維持。
- 学習ループへの SymPy/BFGS 混入はない（データは事前生成、ステップ計時ログでも
  data 時間 ≪ train 時間）。ステップ計時ログは `train_model(log_every_steps=…)` で常設化。

### 5.4 in-distribution 評価（M7 検証）

held-out 100 問（学習と同じ事前分布、シード 999）。
コマンド: `python scripts/eval_indist.py --model runs/proto3var --n 100 --k {32,8}`
所要: k=32 で約 35 分、k=8 で約 13 分（1 問あたりの支配項は BFGS フィット）。

| 指標 | k=32 | k=8 |
|---|---|---|
| 記号的一致率 | **0.16** | 0.14 |
| スケルトン一致率 | 0.10 | 0.10 |
| R²>0.999 率 | 0.27 | 0.22 |
| 平均無効生成率 | 0.32 | 0.35 |
| 候補ゼロ率 | 0.00 | 0.00 |

観察:
- tiny1var（ゲート 7/10）に対し、3変数・演算子10種の式空間では復元率が大きく
  低下（0.16）。式空間の拡大にモデル容量・学習量・k が追いついていない。
- k=8→32 の伸びが +0.02 と小さい。tiny の k 曲線（4→16 で +0.3）と対照的で、
  外部レビュー §8 の「語彙拡大で必要な k が増える」指摘と整合。k=128〜256 級の
  探索か、モデル自体の強化（学習量・d_model）が必要（フェーズ5の課題に記載）。
- 記号的一致（0.16）と R²>0.999（0.27）の差 = 「数値的には合うが別の式」が約1割。
- スケルトン一致率（0.10）< 記号的一致率（0.16）なのは、スケルトン一致が
  「トークン列の完全一致」という厳格判定のため（等価な別構造を含まない）。
